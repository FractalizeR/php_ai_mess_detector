# 06 — HAL 9000

**Persona:** AI coding assistant analyzing architecture via JSON
**Projects:** Doctrine ORM (453 files, 55k LOC), Symfony Console (132 files, 19k LOC)
**Date:** 2026-03-16

## Executive Summary

AIMD's JSON output is **well-structured for programmatic consumption** and provides enough data for an AI agent to generate meaningful architecture assessments. The two-format approach (json for violations+health, metrics-json for raw numbers) covers most analysis needs. Key strengths: health scores with per-dimension breakdown, worst-offender lists, instability/abstractness at namespace level. Key gaps: violation truncation at 50 in base JSON (critical for full analysis), message/recommendation duplication, no dependency graph data, no inter-run diff support. The namespace drill-down is genuinely useful — it removes truncation and focuses the analysis.

## Findings

### Finding H1: Violation list truncated to 50 in base JSON — unusable for full analysis
**Severity:** HIGH
**Category:** missing-data

The base `json` format caps violations at 50 out of 1177 (Doctrine) / 579 (Symfony). An AI agent cannot perform comprehensive analysis with 4% of violations. The `violationsMeta.byRule` provides aggregate counts per rule (useful), but you cannot see which specific classes/methods are affected beyond the top 50. Workaround: use `--namespace` drill-down which returns all violations for the scope. But this requires multiple invocations and prior knowledge of which namespaces to investigate.

**Recommendation:** Add `--limit=N` option to control violation count in JSON output (default 50, allow `--limit=0` for unlimited). Alternatively, add a `violationsByNamespace` or `violationsByRule` grouped structure.

### Finding H2: `message` and `recommendation` fields are always identical
**Severity:** HIGH
**Category:** schema-issue

In 100% of sampled violations (50/50 Doctrine, 50/50 Symfony), `message === recommendation`. This wastes tokens for an AI consumer and provides no additional value. The intent seems to be: `message` describes the problem, `recommendation` suggests a fix. Currently, both contain the same text like "NPath complexity: > 10^9 (threshold: 1000) — explosive number of execution paths".

**Recommendation:** Either (a) populate `recommendation` with distinct actionable advice (e.g., "Extract conditional branches into separate methods" for NPath, "Apply Strategy pattern to reduce coupling" for CBO), or (b) remove the `recommendation` field to reduce payload size. Option (a) is far more valuable for AI consumers.

### Finding H3: No dependency graph data in JSON
**Severity:** HIGH
**Category:** missing-data

Neither `json` nor `metrics-json` format exposes the dependency graph. I can see coupling numbers (Ca, Ce, CBO, instability) but not WHO depends on WHOM. The circular dependency violation messages contain the cycle text ("SQLFilter -> EntityManagerInterface -> FilterCollection -> SQLFilter"), but this is in a human-readable string, not structured data. Without a dependency adjacency list, an AI agent cannot:
- Determine safe refactoring order
- Identify architectural layers
- Suggest where to introduce interfaces for decoupling
- Map the impact radius of a change

**Recommendation:** Add a `dependencies` section to metrics-json with per-namespace or per-class adjacency data: `{"from": "A", "to": "B", "type": "uses"}`. The data is already computed internally (instability = Ce/(Ca+Ce)), so exposing it is low-effort.

### Finding H4: Health decomposition is empty
**Severity:** MEDIUM
**Category:** missing-data

The `health.*.decomposition` array is always `[]` in JSON output. This field appears intended to show what sub-metrics contribute to each health score. Without it, an AI agent sees "complexity health = 0" but cannot explain WHY or which specific metrics pulled it down. The `worstClasses[].metrics` helps partially (showing cbo, ccn.avg, tcc, wmc, mi.avg), but it does not map to health formula inputs.

**Recommendation:** Populate decomposition with formula inputs, e.g., `[{"metric": "ccn.avg", "value": 16.4, "weight": 2.0, "contribution": -32.8}]`. This enables AI agents to explain root causes.

### Finding H5: Tech debt estimates are flat/coarse
**Severity:** MEDIUM
**Category:** schema-issue

Tech debt values in violations are either 30 or 45 minutes — only 2 distinct values across 50 sampled violations. This makes debt data nearly useless for prioritization: a method with CCN=21 (just over threshold) has the same 30-minute debt as NPath > 10^9. An AI agent cannot differentiate effort levels.

**Recommendation:** Scale debt by severity of the overshoot (e.g., `debt = base * log(metricValue / threshold)`). This would give NPath > 10^9 a much higher debt than CCN=21.

### Finding H6: `computed.health` violations lack specificity
**Severity:** MEDIUM
**Category:** ux-issue

The `computed.health` rule generates the most violations (231 in Doctrine, 85 in Symfony) but its messages are generic: "Reduce complexity by extracting methods, simplifying conditional logic..." These do not reference the specific health dimension score or the worst contributing metric. For an AI agent, these violations are noise — the same information is available (more precisely) from `worstNamespaces` and `worstClasses` health scores.

**Recommendation:** Either make health violation messages specific (include the score and dimension), or allow AI consumers to filter them out without losing signal. Adding `"healthDimension": "complexity"` and `"healthScore": 0` to the violation would help.

### Finding H7: Namespace drill-down is very useful
**Severity:** N/A
**Category:** good-structure

The `--namespace=X` flag with JSON output is exactly what an AI agent needs for deep analysis. It returns all violations (not truncated), includes child namespaces in `worstNamespaces`, and provides class-level detail in `worstClasses`. The `directScore` vs recursive `score` in `health.overall` is a smart distinction — it tells you whether the namespace itself is bad or just its children.

### Finding H8: `worstClasses` metrics selection is well-curated
**Severity:** N/A
**Category:** good-structure

The `worstClasses[].metrics` dict includes exactly the right fields for quick assessment: `methodCount`, `propertyCount`, `cbo`, `ccn.avg`, `tcc`, `wmc`, `mi.avg`. This is a minimal but sufficient set for an AI to diagnose the problem (god class? low cohesion? high coupling?). The per-dimension `healthScores` breakdown is also valuable.

### Finding H9: Circular dependency detail is only in message string
**Severity:** MEDIUM
**Category:** schema-issue

Circular dependency violations contain the cycle path only as a human-readable string: "Circular dependency (177 classes): SQLFilter -> EntityManagerInterface -> FilterCollection -> SQLFilter". For an AI agent, this should be structured data: `{"cycle": ["SQLFilter", "EntityManagerInterface", "FilterCollection", "SQLFilter"], "length": 177}`. Parsing natural language is fragile and error-prone.

**Recommendation:** Add a `details` field to violations that can carry rule-specific structured data (cycle path for circular deps, duplication locations for code duplication, etc.).

### Finding H10: No way to get all violations without namespace scoping
**Severity:** MEDIUM
**Category:** ux-issue

The only way to get un-truncated violations is via `--namespace`. But there is no equivalent `--all-violations` or `--limit=0` flag for the base JSON. An AI agent workflow requires: (1) run base JSON for overview, (2) identify worst namespaces, (3) run per-namespace drill-downs. This is reasonable but could be optimized to a single invocation.

### Finding H11: Metrics-json has no violations, JSON has no raw metrics
**Severity:** LOW
**Category:** schema-issue

The two formats are complementary but require two separate tool invocations. The `json` format has violations + health + worst offenders but no raw metrics. The `metrics-json` has all raw metrics but no violations. For an AI agent, a single combined output would be more efficient. However, the current separation is acceptable — the metrics-json is large (3433 symbols for Doctrine) and bloating the violation JSON would hurt smaller use cases.

### Finding H12: `metricValue: null` and `threshold: null` on some violations
**Severity:** LOW
**Category:** schema-issue

Some violations (e.g., `computed.health` for complexity) have `metricValue: 0` and `threshold: null`. This is technically correct (health violations are computed, not threshold-based) but confusing for programmatic consumption. An AI agent filtering by `metricValue > threshold` would fail on null. Consider using a consistent sentinel or omitting these fields when not applicable.

## JSON Schema Assessment

**Good:**
- Top-level structure is clean and predictable: `meta`, `summary`, `health`, `worstNamespaces`, `worstClasses`, `violations`, `violationsMeta`
- Field names are consistent and self-documenting (camelCase throughout)
- `violationsMeta.byRule` provides complete rule-level counts even when violations are truncated — essential for triage
- Health scores use a consistent `{score, label, threshold}` structure
- Violation objects have all necessary fields for programmatic routing: `rule`, `code`, `severity`, `file`, `line`, `symbol`, `namespace`

**Confusing/Missing:**
- The `code` field (e.g., `complexity.npath.method`) vs `rule` (e.g., `complexity.npath`) — the distinction is unclear without documentation. `code` includes the level suffix, `rule` does not
- No JSON Schema definition or OpenAPI spec for the output — an AI agent has to discover the structure empirically
- `worstClasses[].healthScores` uses raw numbers, while `health.*` uses `{score, label}` objects — inconsistent typing
- No `version` field on the violation schema itself (only in `meta`)

## Metrics JSON Assessment

**Granularity:** Excellent. Four levels (project, namespace, class, method) plus file-level LOC data. 3433 symbols for a 453-file project — comprehensive.

**Completeness:** Very good. All major metric families present at every applicable level:
- Method: ccn, cognitive, npath, halstead.*, mi, parameterCount, methodLoc
- Class: cbo, ca, ce, instability, tcc, lcc, lcom, wmc, rfc, dit, noc, classRank, typeCoverage.*, health.*
- Namespace: all class metrics aggregated (sum/avg/max/min/count), plus own abstractness, instability, distance
- Project: full roll-up

**Trend analysis:** Possible but not natively supported. An AI agent would need to:
1. Store two metrics-json outputs
2. Match symbols by `name` (unique per type)
3. Diff metric values

This works but is fragile (renamed classes break matching). A `--diff=previous.json` flag or stable symbol IDs would help.

**Missing metrics:**
- No duplication metrics in metrics-json (code duplication is only in violations)
- No RFC breakdown per method (only class-level aggregate)
- No dependency list (only aggregate Ca/Ce/CBO numbers)

## UX Notes

**Easy:**
- Discovering the structure: top-level keys are self-explanatory
- Identifying worst offenders: `worstNamespaces` and `worstClasses` are pre-sorted and pre-curated
- Health triage: the 6-dimension health model maps cleanly to refactoring categories
- Namespace drill-down: `--namespace` flag works exactly as expected, returns focused data

**Hard:**
- Getting complete violation data: required discovering `--namespace` workaround for truncation
- Understanding health score formulas: no decomposition data means scores are opaque
- Correlating JSON and metrics-json: need to cross-reference by symbol name between two files
- Circular dependency analysis: had to parse human-readable strings
- Understanding the `directScore` vs `score` distinction: not documented in the output itself

**Token efficiency:** The base JSON for Doctrine is ~50KB (50 violations) — reasonable. Metrics-json is ~1.5MB (3433 symbols) — large but manageable. For an AI agent with limited context, the violation JSON is the right entry point; metrics-json should be queried selectively.

## Guide Notes

**Recommended AI Agent Workflow for Architecture Analysis:**

1. **Overview scan:** `bin/aimd check <path> -f json --workers=0` — get health scores, worst offenders, violation distribution
2. **Identify hotspots:** Sort `worstNamespaces` by `healthOverall`. Focus on namespaces with health < 50 (Weak/Critical)
3. **Deep dive:** For each hotspot namespace: `bin/aimd check <path> --namespace=<ns> -f json --workers=0` — get all violations, class-level detail
4. **Raw metrics (optional):** `bin/aimd check <path> -f metrics-json --workers=0` — for custom analysis, instability/abstractness mapping, trend comparison
5. **Generate recommendations:** Cross-reference health dimensions with specific violations. Map coupling issues to instability/abstractness. Use `worstClasses[].metrics` for sizing refactoring effort

**Most useful fields for AI agents:**
- `health.*` — quick project health assessment
- `worstNamespaces[].healthScores` — identifies which dimension is worst per namespace
- `worstClasses[].metrics.{cbo, tcc, wmc, mi.avg}` — class-level diagnosis
- `violationsMeta.byRule` — violation distribution without needing individual violations
- `violations[].{code, metricValue, threshold}` — programmatic severity assessment
- Namespace drill-down `health.overall.directScore` — distinguishes "this namespace is bad" from "its children are bad"

**Data gaps to work around:**
- Violation truncation: always use `--namespace` for complete data
- No dependency graph: infer direction from Ca/Ce ratios and instability
- Duplicate message/recommendation: ignore `recommendation` field
- Flat tech debt: use `metricValue / threshold` ratio as severity proxy instead
- No trend support: store metrics-json outputs and diff by symbol name externally
