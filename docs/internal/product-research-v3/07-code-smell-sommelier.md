# 07 — Code Smell Sommelier

**Persona:** QA lead assessing code smell and security detection accuracy
**Projects:** PHP-Parser (269 files), Doctrine ORM (453 files), Laravel (1536 files), Flysystem (55 files)
**Date:** 2026-03-16

## Executive Summary

Tested all 17 code smell rules and 5 security rules across four PHP projects of varying quality and size. Overall, the detection is solid with well-calibrated thresholds for most rules. Key issues:

1. **identical-subexpression** has a catastrophic false positive problem in generated code (782/782 in PHP-Parser are FPs from YACC-generated parser files)
2. **debug-code** conflates library API methods (`dd()`, `dump()`, `var_export()`) with accidental debug leftovers — ~80% FP rate in Laravel
3. **data-class** over-flags exception classes and interfaces — ~40-60% FP rate depending on project
4. **boolean-argument** is noisy in framework code but honest about the smell — ~30% FP in application code, ~60% in frameworks
5. **Security rules** are conservative and trustworthy — 0 false positives for sensitive-parameter, 2 FPs for hardcoded-credentials
6. **Message quality** is inconsistent — some rules provide rich detail (god-class, data-class, long-parameter-list), others are bare minimum (debug-code, empty-catch, boolean-argument lack `humanMessage` and `recommendation`)

Rules that **never fired** across all 4 projects: unreachable-code, sql-injection, xss, command-injection. This could indicate either excellent code quality in the test projects or overly conservative detection.

## Findings

### Finding CS1: identical-subexpression has catastrophic FP on generated code
**Severity:** HIGH
**Category:** false-positive

All 782 identical-subexpression violations in PHP-Parser come from auto-generated YACC parser files (`Php7.php`, `Php8.php`). The pattern flagged is `$stackPos-(1-1)`, `$stackPos-(2-2)`, etc. — arithmetic expressions where both sides of subtraction are the same number. These are generated by the parser generator and are completely intentional.

**Impact:** This single rule accounts for 90% of all violations in PHP-Parser, making the overall smell profile wildly misleading. The 865 total violations drop to 83 without this rule.

**Recommendation:** Consider either (a) excluding auto-generated files by detecting `@generated` annotations or characteristic patterns, or (b) adding documentation guidance about using `--exclude` for generated files. This is a documentation/UX issue more than a detection bug — the rule correctly identifies identical subexpressions, but the context makes them benign.

### Finding CS2: debug-code flags intentional framework API methods
**Severity:** HIGH
**Category:** false-positive

The `debug-code` rule flags `dd()`, `dump()`, `var_export()`, and `debug_backtrace()` as debug leftovers. In Laravel, the majority of findings are **intentional public API methods**:

- `Dumpable::dd()` and `Dumpable::dump()` — a trait used across the framework as a developer convenience API
- `Builder::dump()`, `Builder::dd()`, `Builder::dumpRawSql()` — query debugging API (4 findings in one file)
- `Response::dump()`, `Response::dd()` — HTTP client debugging API
- `TestResponse::dump()` etc. — testing utilities

In Doctrine, `var_export()` in `Node::__toString()` and `Debug::dump()` are legitimate library code.

In Flysystem, `var_export()` in `InvalidVisibilityProvided::withVisibility()` is used for error message formatting — clearly not debug code.

Estimated: ~35/47 Laravel findings (~75%) and 4/5 Doctrine findings (~80%) are false positives. Only cases where `dump()`/`dd()` appear inside non-dump methods would be true positives.

**Recommendation:** The rule needs contextual awareness. Possible improvements:
1. Don't flag `var_export($x, true)` (return mode, not output mode)
2. Don't flag when the function call is inside a method named `dump`, `dd`, `debug`, etc.
3. Don't flag `debug_backtrace()` (it's used for stack trace analysis, not debugging)
4. Consider lowering severity from ERROR to WARNING

### Finding CS3: data-class over-flags exceptions and interfaces
**Severity:** MEDIUM
**Category:** false-positive

In Flysystem, 16 of 18 data-class findings are on exception classes (`UnableToReadFile`, `UnableToCopyFile`, etc.) and interfaces (`FilesystemReader`, `StorageAttributes`). Exception classes inherently have high WOC (all methods are public) and low WMC — they are DTOs by design. Flagging them as "Data Classes" with the suggestion to "Consider encapsulating behavior" is unhelpful.

In Laravel, `AuthorizationException`, `AuthenticationException`, and similar classes are flagged. Service provider classes like `ViewServiceProvider` are also flagged (WOC=89%, WMC=9).

**Recommendation:**
1. Exclude classes extending `\Exception` / `\RuntimeException` / `\InvalidArgumentException` (and similar) from data-class detection
2. Exclude interfaces (they have 100% WOC by definition)
3. The `excludeReadonly` option exists but doesn't cover these cases

### Finding CS4: boolean-argument is honest but noisy in frameworks
**Severity:** MEDIUM
**Category:** threshold-issue

In Laravel (90 findings), common patterns flagged:
- `$force` (5 occurrences) — used across cache, auth, symlinks
- `$strict` (5) — comparison mode toggles
- `$condition` (5) — conditional execution flags
- `$always` (4) — "always do X" toggle
- `$newline` (3) — output formatting

Many of these ARE genuine boolean-argument smells by the book (Martin Fowler would agree). However, in framework code, these are pragmatic API design choices. The `$default` parameter in `Arr::boolean(array, key, ?bool $default = null)` is a borderline case — it's a default value, not a behavioral flag.

No `?Type $param = null` patterns were incorrectly flagged as boolean arguments — the rule specifically targets `bool` type hints, which is correct.

**Recommendation:** The detection is accurate per the definition. The FP rate depends on context:
- Application code: ~30% FP (most boolean args are genuine smells)
- Framework/library code: ~60% FP (pragmatic API design)
- Consider providing guidance in documentation about expected noise levels in library vs application code

### Finding CS5: empty-catch detects legitimate chain-of-responsibility patterns
**Severity:** MEDIUM
**Category:** false-positive

In Flysystem, the one empty-catch finding in `ChainedPublicUrlGenerator` is a textbook chain-of-responsibility pattern — try each generator, catch failure, move to next. This is intentional and well-structured code:

```php
foreach ($this->generators as $generator) {
    try {
        return $generator->publicUrl($path, $config);
    } catch (UnableToGeneratePublicUrl) {
    }
}
```

In Laravel, `DatabaseStore::add()` catches `QueryException` with an empty block as a "insert or ignore" pattern for SQL Server. The `CompiledRouteCollection` catches exceptions in route matching. These are ~40% FP.

**Recommendation:** The rule could reduce FPs by:
1. Not flagging empty catches inside `foreach` loops (chain pattern)
2. Accepting catches with a comment (e.g., `// ...` as in Laravel's DatabaseStore) as "acknowledged"
3. Current behavior of flagging all empty catches is correct for the strictest interpretation

### Finding CS6: sensitive-parameter security findings are excellent
**Severity:** LOW (positive finding)
**Category:** useful-detection

All 27 sensitive-parameter findings in Laravel are on methods handling `$credentials`, `$password`, `$secret`, or `$token` parameters. Examples:
- `SessionGuard::once(array $credentials)` — 9 findings covering attempt, validate, login methods
- `PendingRequest::withBasicAuth($username, $password)` — correctly identifies auth methods
- `MemcachedConnector::connect()` — correctly identifies server connection with credentials

These are genuine security improvements. Adding `#[\SensitiveParameter]` to these methods would prevent credential leakage in stack traces and error logs.

The `PreventRequestsDuringMaintenance::bypassResponse(string $secret)` finding is borderline — the maintenance bypass secret is less sensitive than authentication credentials, but still worth protecting.

**Estimated FP rate: ~5%** (only the maintenance bypass is questionable).

### Finding CS7: hardcoded-credentials flags translation strings
**Severity:** MEDIUM
**Category:** false-positive

Both hardcoded-credentials findings in Laravel are in translation files:
- `auth.php` line 17: `'password' => 'The provided password is incorrect.'` — the word "password" as an array key in a language file
- `validation.php` line 39: `'current_password' => 'The password is incorrect.'` — same pattern

These are 100% false positives. The rule is detecting the word "password" as an array key and assuming it's a hardcoded credential.

**Recommendation:** The rule should exclude `.php` files under common translation/lang directories, or require that the **value** (not key) matches credential-like patterns (e.g., secrets, tokens, not human-readable messages).

### Finding CS8: unused-private misses trait method calls
**Severity:** MEDIUM
**Category:** false-positive

In Doctrine ORM, `XmlDriver::doLoadMappingFile()` is flagged as unused-private, but it IS called from the `LoadMappingFileImplementation` trait that the class uses. The tool doesn't resolve trait method calls to the composing class.

The other 2 findings (`ProxyFactory::generateUseLazyGhostTrait()` and `ProxyFactory::generateSerializeImpl()`) appear to be genuine dead code — no references found anywhere in the Doctrine ORM codebase.

**FP rate: 33%** (1/3 findings is a trait-related false positive).

### Finding CS9: superglobals findings are mostly legitimate
**Severity:** LOW
**Category:** useful-detection

All 36 superglobal findings in Laravel are in foundation/infrastructure code:
- `Application.php` (11 findings) — accessing `$_SERVER`, `$_ENV` for environment detection
- `Cloud.php` (6) — checking cloud environment variables via `$_SERVER`
- `resources/server.php` (4) — built-in development server

These are appropriate places for superglobal access in a framework. The rule correctly doesn't fire in Flysystem, Doctrine, or PHP-Parser (which don't touch superglobals). The findings are informative even if not actionable for framework code — they help identify where environment coupling exists.

### Finding CS10: god-class detection is well-calibrated
**Severity:** LOW (positive finding)
**Category:** useful-detection

The god-class rule produces highly actionable findings:
- PHP-Parser: `ParserAbstract` (WMC=245, LCOM=9, 1297 LOC), `PrettyPrinterAbstract` (WMC=254, LCOM=3, TCC=0.17, 1685 LOC) — both are legitimately complex but unavoidable for a parser
- Doctrine: `UnitOfWork` (WMC=478, TCC=0.27, 3199 LOC) — the classic ORM god class
- Laravel: 103 findings, but `Factory` (WMC=51, LCOM=3, TCC=0.12, 631 LOC), `Builder` classes — genuinely over-responsible

The 3/4 and 4/4 criteria matching with specific numbers is very helpful for prioritization. ERROR severity for 4/4 criteria match vs WARNING for 3/4 is a good distinction.

### Finding CS11: Three security rules never fire
**Severity:** MEDIUM
**Category:** missing-detection

`security.sql-injection`, `security.xss`, and `security.command-injection` produced zero findings across all four projects (2313 files total). While these projects use abstractions (query builders, template engines) that prevent most injection, it's worth investigating:

1. **Doctrine ORM** generates SQL queries — some DQL-to-SQL translation involves string concatenation. Does the rule catch `$conn->executeQuery("SELECT ... " . $userInput)`?
2. **Laravel** has `DB::raw()` usage patterns and Blade `{!! $var !!}` — are these detected?
3. Zero findings could indicate the rules are working correctly (these frameworks abstract away raw SQL/HTML) OR the rules are too conservative.

Without access to projects with known vulnerabilities, hard to determine if these rules would catch real issues.

### Finding CS12: long-parameter-list threshold is reasonable
**Severity:** LOW (positive finding)
**Category:** useful-detection

Warning threshold at 4, error at 6. In practice:
- Flysystem `FileAttributes::__construct(6 params)` — correctly flagged as error
- Doctrine `SchemaTool::gatherRelationJoinColumns(7 params)` — legitimate finding
- Constructor DI is flagged, but the separate `constructor-overinjection` rule handles that with its own thresholds

The message "consider introducing a parameter object" is actionable. No issues observed with DI constructors being double-flagged (constructor-overinjection has its own, higher threshold).

### Finding CS13: eval findings are all genuine
**Severity:** LOW (positive finding)
**Category:** useful-detection

Doctrine's `eval(substr($proxyCode, 5))` in ProxyFactory — generating proxy classes at runtime. Laravel's eval — same pattern. These are real eval usage that represents a known security/maintenance tradeoff. The findings are correct and the severity (ERROR) is appropriate.

## Security Rules Assessment

**Trust level: MEDIUM-HIGH for sensitive-parameter, LOW for hardcoded-credentials, UNKNOWN for injection rules.**

**sensitive-parameter** is the standout security rule. It correctly identifies 27 methods in Laravel that handle credentials without `#[\SensitiveParameter]`, and all findings are actionable. This rule alone justifies enabling security scanning.

**hardcoded-credentials** needs refinement. The 2 findings are both false positives (translation file keys). The rule appears to trigger on array keys containing words like "password" regardless of context.

**sql-injection, xss, command-injection** never fired. Without testing against deliberately vulnerable code, the effectiveness is unknown. For library code that uses abstractions, zero findings is expected and correct. But users might have false confidence that their code is injection-free when these rules simply don't detect complex patterns.

**Would I trust these in a security review?** I'd trust sensitive-parameter completely. I'd suppress hardcoded-credentials findings after manual review. I would NOT rely on the injection rules as a substitute for security auditing — they should be positioned as "catch obvious mistakes" rather than "comprehensive security scan."

**Key message quality issues:**
1. **No humanMessage** for any code smell or security rule — all return N/A. This is a gap if the summary view relies on humanMessage for context.

## UX Notes

1. **Violation density is overwhelming in large projects.** Laravel produces 1060 code smell violations. Without `--only-rule` filtering, users will be buried. The summary format helps, but the `--detail` output needs pagination or rule-level grouping to be practical.

2. **The `--only-rule=code-smell` prefix matching works well.** Being able to filter to all code smells or a specific one (`code-smell.boolean-argument`) is intuitive. Same for `--only-rule=security`.

3. **Missing: severity filtering in --only-rule.** Users might want `--only-rule=code-smell --fail-on=error` to ignore warnings. This is available via `--fail-on` but it's not obvious from the help text.

4. **The technical debt summary by rule is very useful.** Seeing "code-smell.identical-subexpression ~24d 3h 30min (782 violations)" immediately tells you where the noise is.

5. **Generated code detection would be valuable.** A `--exclude-generated` flag or heuristic (file comment contains `@generated`, `This file is auto-generated`, etc.) would eliminate the biggest source of false positives.

6. **data-class excludeReadonly and excludePromotedOnly options are good** but don't cover the exception/interface case. An `excludeExceptions` option would help.

## Guide Notes

### How to interpret code smell findings

**Act immediately (high signal, low FP):**
- `eval` — always investigate. Even legitimate uses should be documented
- `goto` — rare and usually refactorable
- `count-in-loop` — easy fix, real performance impact
- `constructor-overinjection` — signals design problems
- `sensitive-parameter` — easy, high-value security improvement

**Configure thresholds for your context:**
- `long-parameter-list` — default 4/6 is reasonable; consider 5/8 for legacy code
- `god-class` — the 3/4 criteria model is good; review WARNING-level findings quarterly
- `boolean-argument` — expect ~50% FP in framework code; lower in application code

**Suppress or disable in specific contexts:**
- `debug-code` — suppress in files that intentionally provide dump/debug API. Consider disabling for `Testing/` directories
- `data-class` — suppress for exception classes and DTOs (use `@aimd-ignore code-smell.data-class`)
- `identical-subexpression` — exclude generated files via `--exclude` or baseline
- `empty-catch` — suppress for chain-of-responsibility patterns; always add a comment explaining why the catch is empty

**Monitor but don't block CI:**
- `superglobals` — informational in framework code, actionable in application code
- `error-suppression` — often necessary for file operations; review case-by-case
- `exit` — legitimate in CLI entry points; problematic in library code

**Security rules:**
- Enable `sensitive-parameter` unconditionally — high signal, low noise
- `hardcoded-credentials` needs manual triage — expect FPs in translation/config files
- Injection rules (sql, xss, command) are supplementary — do not rely on them as primary security scanning
